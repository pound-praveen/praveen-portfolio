Context

The pincode problem

Nilkamal Furniture delivers to 19,000 of approximately 22,000 pincodes across India — covering 87% of serviceable locations. Yet every user, regardless of whether their pincode was serviceable, was required to manually enter and validate it before they could add any product to their cart.

This mandatory gate — placed right at the top of the checkout funnel — created significant friction for the vast majority of users who were already in a serviceable zone. The system was built to catch the 13% edge case, but it was penalising the 87% majority every single session.

Friction 01
No input limit enforcement
The pincode field accepted more than 6 digits with no validation cap, causing the API to fire on invalid inputs and returning confusing results.
Friction 02
Premature API calls
The serviceability check triggered on every keystroke, including inputs beyond 6 digits — wasting API calls and degrading the experience.
Friction 03
Error hidden on mobile
When users tapped "Add to Cart" without entering a pincode, the error message rendered off-screen on mobile devices — invisible, with no scroll nudge.
Friction 04
Unclear delivery information
Delivery details were vague — no specific delivery date, no free-delivery callout — reducing user confidence and increasing pre-checkout drop-off.
Before & After

The interaction, redesigned

The core change was shifting from a friction-first model (user must act before the system helps) to an assistance-first model (the system pre-fills what it knows, and steps aside).

Before — friction at every step
Pincode
Please enter a valid 6-digit pincode
No input limit — 7+ digits accepted silently
API fires on invalid inputs
User must manually delete & retype to correct
No default pincode — every user starts from zero
After — assistance first
Pincode
Delivery by Thursday, 9 Jan  FREE
Default pincode 400028 (Mumbai) pre-filled
API auto-triggers on 6th digit — no button needed
Clicking the field selects all text instantly
Specific delivery date shown with free delivery badge
Solution

Four changes, one clear goal

Each solution mapped directly to a friction point identified during usability testing. The approach was minimal intervention with maximum funnel impact.

Auto-trigger on 6 digits
The serviceability API fires automatically once the 6th digit is entered — eliminating the need for a separate "Check" button tap.
Default pincode: 400028
Pre-filling Mumbai's central pincode reduces the funnel's first step from a required action to an optional correction.
Clearer delivery details
Delivery information now shows the specific day and date, with a prominent "Free Delivery" badge wherever applicable.
Select-all on input focus
Tapping the pincode field selects the entire pre-filled value — users can immediately type without manually deleting each digit.

Next Steps

The Bengaluru hypothesis

The default pincode of 400028 (Mumbai) was chosen for the initial rollout as a pragmatic baseline. However, session data reveals a compelling opportunity worth testing.

Dark = Nov–Dec 2025  ·  Light = Oct–Nov 2025

⚑ Proposed experiment
Bengaluru now holds the highest session volume across all cities and is on an upward trend (+9%), while Mumbai is essentially flat. The hypothesis: switching the default pincode from 400028 (Mumbai) to 560001 (Bengaluru) will further reduce friction for the single largest user cohort. Recommended: A/B test with a 50/50 split across both default pincodes, measured over a 30-day window using the same funnel metrics.
